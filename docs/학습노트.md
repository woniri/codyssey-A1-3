# 학습노트 — 타래(Tarae) 프로젝트로 이해하는 웹 개발 기초

> 이 문서는 "정답 암기"가 아니라 "이해"를 목표로 해요. 여러분이 실제로 만든 타래 프로젝트의 코드를 예시로 써서
> 하나씩 설명할게요. 순서대로 읽으면 과제 목표 6가지와 평가문항 전체를 자연스럽게 이해할 수 있어요.

---

## 0. 전체 그림 먼저 보기

타래가 동작하는 순서를 아주 단순하게 그리면 이래요.

```
[사용자가 브라우저에서 클릭/입력]
        ↓
[JavaScript가 그 행동을 감지]
        ↓
[필요하면 서버(api/)에 요청 전송]
        ↓
[서버가 처리(예: AI 호출) 후 결과 반환]
        ↓
[JavaScript가 받은 결과를 화면에 그려줌]
```

이 문서는 이 화살표 하나하나를 설명하는 문서예요.

## 1. HTML / CSS / JavaScript, 각각 뭘 하는 걸까?

**비유: 집을 짓는다고 생각해보세요.**

| | 집으로 비유하면 | 타래 코드로 보면 |
|---|---|---|
| **HTML** | 뼈대(기둥, 벽, 문의 위치) — "여기에 방이 있다"만 정함 | `index.html`의 `<input id="thought-input">` → "여기 입력창이 있다"만 알려줌. 이걸 눌렀을 때 뭐가 일어나는지는 전혀 모름 |
| **CSS** | 인테리어(벽지 색, 가구 배치) — "어떻게 보이는가" | `css/style.css`의 `--accent-color: #4a6fa5;` → 버튼 색을 파란 계열로 통일. `dark-mode.css`는 같은 화면인데 색만 어둡게 바꿔주는 "조명 스위치" 역할 |
| **JavaScript** | 사람이 실제로 움직이는 것(문을 열고, 스위치를 누르고) — "동작" | `js/stream.js`가 "던지기" 버튼 클릭을 감지해서 → 입력값을 읽고 → 저장하고 → 버튼 색을 초록으로 바꿈 |

**핵심 포인트**: HTML만 있으면 화면은 보이지만 버튼을 눌러도 아무 일도 안 일어나요. CSS만 있으면 예쁘지만 역시 안 움직여요.
**JavaScript가 있어야 "버튼을 눌렀을 때 실제로 뭔가 일어나는" 상호작용이 생겨요.**

이게 왜 미션 4번 요구사항의 "AI 기능 UX 최소 기준"이 항상 JavaScript 파일에서 처리되는지 이유예요.

## 2. 사용자 입력이 fetch를 통해 요청이 되고, 응답이 화면에 반영되는 흐름

**비유: 식당에서 주문하기**

1. 손님(사용자)이 메뉴판(화면)을 보고 "이거 주세요" 라고 말함 (버튼 클릭)
2. 종업원(JavaScript)이 그 주문을 주방(서버)에 전달함 → 이게 `fetch()`
3. 주방(서버, `api/` 폴더)이 요리를 만듦 → 이게 Python 서버리스 함수가 하는 일
4. 종업원이 완성된 요리를 손님 테이블로 가져다줌 → 이게 서버의 응답(response)
5. 손님이 요리를 받아봄 → 화면에 결과가 표시됨

**실제 코드로 보면** (`vault.html`의 AI 기능 부분):

```javascript
const response = await fetch("/api/expand", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ content: "사용자가 고른 생각 2개" })
});
const json = await response.json();  // 주방에서 온 요리(결과) 받기
draftText = json.expansion;           // 요리를 접시에 담기(화면에 표시)
```

- `fetch("/api/expand", ...)` = "api/expand라는 이름의 주방에 이 재료(데이터)를 갖고 가서 요리해줘"
- `await` = 요리가 완성될 때까지 기다린다는 뜻 (기다리는 동안 로딩 문구를 보여주는 이유가 이거예요)
- `response.json()` = 완성된 요리를 받아서 우리가 알아볼 수 있는 형태(JSON)로 꺼냄

## 3. Vercel Serverless Functions가 뭐고, 왜 쓰는가?

**비유: 24시간 켜져 있는 주방이 아니라, 주문 들어올 때만 나타나는 아르바이트 요리사**

일반적인 서버는 컴퓨터 한 대를 항상 켜놔야 해요 (전기세 항상 나감).
**Serverless Function**은 "요청이 올 때만 잠깐 실행되고, 끝나면 사라지는" 방식이에요.
그래서 아르바이트생이 손님 주문 있을 때만 나타나서 요리하고, 없으면 아예 안 나타나는 것과 비슷해요. → 안 쓸 때는 비용이 안 들어요.

타래에서는 `api/` 폴더 안의 각 `.py` 파일 하나하나가 "아르바이트 요리사 한 명"이에요.

```python
class handler(BaseHTTPRequestHandler):
    def do_POST(self):
        # 요청이 들어왔을 때만 이 코드가 실행됨
        ...
```

- `api/expand.py` = 아이디어 확장 전담 요리사
- `api/classify.py` = 태그 분류 전담 요리사
- `api/config.py` = 안전한 설정값만 살짝 알려주는 요리사

**프론트에서 백엔드를 부르는 구조**: 프론트(`vault.html`)는 `/api/expand`라는 "주소"만 알고 있으면 돼요.
실제로 그 주소 뒤에서 Python 코드가 돌고 있다는 사실은 브라우저 입장에서는 몰라도 상관없어요 — 이게 프론트와 백엔드가 분리되는 이유예요.

## 4. 환경 변수로 API 키를 안전하게 관리해야 하는 이유

**비유: 우리 집 현관 비밀번호를 대문에 매직으로 써 붙이면 안 되겠죠?**

`GEMINI_API_KEY`, `SUPABASE_SERVICE_ROLE_KEY` 같은 값은 이 키를 가진 사람은 누구나
- 여러분 대신 AI를 무제한으로 호출해서(→ 과금 폭탄)
- 여러분의 데이터베이스를 마음대로 읽고 지울 수 있어요(→ 개인정보 유출)

만약 이 키를 코드 안에 직접 써서 GitHub에 올리면, **저장소가 공개되는 순간 전 세계 누구나 그 비밀번호를 볼 수 있어요.**
(비공개 저장소여도 실수로 공개 전환하거나, 캡처/공유하다가 새어나가는 경우가 흔해요.)

그래서 이렇게 나눠요:
- 코드에는 **키 이름만** 적음: `os.environ.get("GEMINI_API_KEY")` — "이 이름의 열쇠를 나중에 줄게" 라는 약속만 코드에 남김
- 진짜 열쇠(값)는 **Vercel 서버 안 금고**(환경 변수 설정)에만 저장 → 코드와 완전히 분리됨

이래서 `.gitignore`에 `.env`가 들어있는 거예요 — 실수로라도 진짜 열쇠 파일이 GitHub에 올라가지 않게 미리 막아두는 장치예요.

## 5. 로컬 환경과 배포 환경의 차이, 그리고 문제 수정 흐름

| | 로컬(내 컴퓨터) | 배포(Vercel, 실제 URL) |
|---|---|---|
| 주소 | `localhost:3000` (나만 접속 가능) | `think-tarae.vercel.app` (전 세계 누구나 접속 가능) |
| 환경 변수 | 내 컴퓨터의 `.env` 파일 | Vercel 대시보드에 등록한 값 |
| 문제 생겼을 때 | 터미널 화면에 에러가 바로 보임 | 브라우저 개발자도구(F12)나 Vercel 로그를 봐야 보임 |

**문제가 생겼을 때 확인하는 순서** (비유: 병원 진료처럼 "어디가 아픈지"부터 좁혀가기):

1. 브라우저에서 **F12**를 눌러서 Console 탭을 봄 → 빨간 에러 메시지가 있는지 확인
2. Network 탭에서 `/api/...` 요청이 빨간색(실패)인지 확인 → 몇 번(404, 500 등) 에러인지 봄
   - 404 = "그 주소 자체가 없다" (파일 경로가 틀렸을 가능성)
   - 500 = "서버 코드 안에서 에러가 났다" (Python 코드 확인 필요)
3. Vercel 대시보드 → 해당 배포 → Functions 로그에서 서버 쪽 에러 메시지 확인
4. 원인을 찾았으면 **새 브랜치**를 만들어 그 파일만 수정
5. Vercel이 자동으로 만들어주는 **Preview 주소**에서 먼저 테스트 (진짜 운영 주소는 안 건드림)
6. 문제없으면 `main`으로 합치기(merge) → 그제서야 진짜 서비스에 반영됨

## 6. AI 코딩 도구가 만든 코드에서 오류를 스스로 파악하는 법

AI가 코드를 대신 써줘도, "이 코드가 왜 이렇게 동작하는지"는 스스로 설명할 수 있어야 해요. 확인하는 순서:

1. **에러 메시지를 있는 그대로 읽기** — 대부분 어디서(몇 번째 줄) 뭐가 문제인지 알려줘요.
2. **그 줄 주변 코드가 무슨 일을 하려던 건지 추측** — 예: `TaraeStorage.getThoughts()`가 `undefined`를 반환했다면 → "데이터를 아직 못 가져왔나?"
3. **AI 도구에게 물어볼 때는 "고쳐줘"보다 "이 에러가 왜 났는지 설명해줘"를 먼저 요청** — 원인을 이해한 다음에 고쳐야 다음에 비슷한 문제가 생겨도 스스로 해결할 수 있어요.
4. 고친 후에는 **같은 상황을 다시 재현**해서 진짜 해결됐는지 확인 (재현 안 되면 우연히 넘어간 것일 수도 있음).

## 7. 실전에서 겪은 문제들 (이번 프로젝트 실제 사례)

이론만 아는 것과 실제로 겪어보는 건 달라요. 이번에 실제로 만났던 문제들을 순서대로 정리했어요.

### 사례 1: 파일이 엉뚱한 곳에 있어서 생긴 404

`js/main.js`를 불러와야 하는데 실제 내용은 `api/main.js`에 있었어요. 브라우저는 있지도 않은 파일을 찾다가 조용히 포기했고, 그 파일이 하던 일(대시보드 숫자 표시)이 통째로 멈춰있었어요.
→ **배운 점**: 화면에 에러가 안 보인다고 문제가 없는 게 아니에요. F12 Console을 꼭 확인하는 습관이 필요해요.

### 사례 2: 프론트와 백엔드의 "약속"이 어긋난 경우

타래장 화면은 생각 2개를 조합하는 기능인데, 정작 호출하던 백엔드 함수는 "생각 1개를 할 일 목록으로 바꿔주는" 전혀 다른 기능이었어요. 둘 다 코드는 멀쩡했지만, **서로 주고받기로 한 데이터 형태(입력/출력 스키마)가 안 맞아서** 실패했어요.
→ **배운 점**: 프론트와 백엔드는 계약서를 주고받는 관계예요. "나는 이런 걸 보낼 테니, 너는 이런 형태로 돌려줘"라는 약속이 어긋나면, 둘 다 잘 짜여진 코드여도 결과는 실패해요.

### 사례 3: 환경 변수는 환경마다 따로 관리된다

Vercel에 배포된 사이트는 잘 되는데, 내 컴퓨터(로컬)에서는 같은 기능이 500 에러를 냈어요. 원인은 Vercel의 환경 변수가 **Production(실제 배포용)/Preview(테스트 배포용)/Development(내 컴퓨터용)**로 따로 나뉘어 있는데, Production에만 등록해뒀던 거였어요.
→ **배운 점**: "환경 변수를 등록했다"는 말이 항상 "모든 곳에서 쓸 수 있다"는 뜻은 아니에요. 어느 환경을 위한 건지 항상 확인해야 해요.

### 사례 4: 데이터베이스는 테이블만 만든다고 끝이 아니다

새 테이블을 만들고 저장을 시도했더니 "403 Forbidden"이 떴어요. 원인은 **RLS(Row Level Security)** 라는 보안 장치였어요 — Supabase는 기본적으로 "이 데이터를 누가 읽고 쓸 수 있는지"를 명시적으로 허용해줘야만 동작해요.
→ **배운 점**: 비유하면, 새로 지은 방에 문은 달았지만 아직 "누가 들어갈 수 있는지" 정하지 않은 상태예요. 방 주인(로그인한 사용자)이 자기 방에 들어가려면, 그 규칙(정책)을 따로 정해줘야 해요.

### 사례 5: 브라우저 보안 정책(CORS)

디스코드/슬랙로 테스트 메시지를 보내려는데 "Load failed"라는 에러만 뜨고 이유를 알 수 없었어요. 알고 보니 브라우저가 보안상 "다른 도메인으로의 직접 요청"을 막고 있었던 거예요(CORS 정책).
→ **배운 점**: 브라우저에서 외부 서비스로 직접 요청하는 게 항상 되는 건 아니에요. 이럴 땐 내 서버를 한 단계 거쳐서(프록시) 요청하면, 서버끼리는 이런 제약이 없어서 문제없이 통신할 수 있어요.
