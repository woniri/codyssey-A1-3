## 항목 2

### Q. html/css/js/api 구조를 왜 이렇게 나눴는가?

- **HTML**은 화면의 뼈대(4개 페이지: `index.html`, `loom.html`, `vault.html`, `settings.html`)만 담당하고, 모든 페이지가 같은 `<nav>` 구조를 공유해 일관된 이동 경험을 주도록 했다.
- **CSS**는 `style.css`(공통 레이아웃/색상 변수)와 `dark-mode.css`(다크모드 전용 오버라이드)로 나눠, 라이트/다크 테마 전환이 CSS 변수(`--bg-color` 등)만 바꿔도 전체 화면에 반영되게 했다.
- **JS**는 역할별로 분리했다: `storage.js`(Supabase 클라이언트 초기화 및 데이터 CRUD를 한 곳에 모은 `TaraeStorage` 객체), `app.js`(다크모드 토글, 로그인 세션 가드처럼 모든 페이지 공통 로직), 각 페이지 전용 로직(예: 홈의 생각 저장)은 별도 파일로 분리해 한 파일이 너무 커지지 않게 했다.
- **api/**는 프론트와 물리적으로 분리해, 브라우저에 노출되면 안 되는 로직(Gemini API 키 사용, Supabase service role key 사용)을 서버에서만 실행되게 격리했다. 기능 단위(분류/조합/확장/일일 인사이트/주간 리포트)로 파일을 쪼개 각 서버리스 함수가 하나의 책임만 갖도록 했다.

### Q. AI 기능의 프론트 로직에서 "로딩/성공/실패" 상태를 어떻게 처리했는가?

`vault.html`의 실타래 연금술(`triggerIdeaAlchemy`) 기준:
- **로딩**: 결과 영역을 먼저 열고 "타래 내부 다중 전문가 패널을 소집하여 생각을 재조합하는 중..." 문구를 즉시 표시한 뒤 `fetch`를 호출한다 (사용자가 대기 상태임을 알 수 있게).
- **성공**: `response.ok`이면 JSON을 파싱해 `expansion`/`text`/`content` 중 있는 필드를 결과 텍스트로 사용해 화면에 표시한다.
- **실패**: 현재는 `catch`에서 별도 안내 없이 대체 텍스트로 넘어가도록 되어 있다 — 이 부분은 "사용자에게 실패를 알리는 안내 문구"로 바꾸는 개선이 필요하다고 스스로 파악하고 있다(디버깅 포인트로 준비해두면 좋음).

### Q. Serverless Function(Python)에서 입력 검증과 응답 포맷을 어떻게 정했는가?

- 모든 API는 `http.server.BaseHTTPRequestHandler`를 상속한 `handler` 클래스 + `do_POST`/`do_GET`로 구현했다(Vercel Python 런타임 규격).
- 입력은 `Content-Length`만큼 body를 읽어 JSON으로 파싱하고, 필요한 키가 없으면 빈 문자열(`.get("content", "")`)로 기본값을 준다.
- 응답은 항상 JSON + `Content-type: application/json`으로 통일했고, 에러가 나도 `try/except`로 감싸 500 상태코드와 `{"error": "..."}` 형태로 응답하도록 해 프론트가 항상 JSON을 파싱할 수 있게 보장했다.
- Gemini 응답은 자유 텍스트로 올 수 있어, 프롬프트 안에 "반드시 이 JSON 포맷으로만 응답해"라고 명시하고 코드에서도 백틱(` ```json `) 마크다운 표시를 제거한 뒤 `json.loads`로 파싱한다.

### Q. 배포 후 발생한 문제를 어떤 순서로 진단하고 수정했는가?

1. 배포 URL 접속 후 브라우저 개발자도구(F12) → Console/Network 탭에서 에러 확인 (404, CORS, 500 등)
2. Vercel 대시보드의 해당 배포 → **Functions 로그**에서 서버리스 함수 쪽 에러 스택 확인
3. 원인이 로컬에서 재현되면 로컬(`vercel dev`)에서 먼저 수정 → 안 되면 원인 파일(프론트 JS/서버 Python) 특정
4. 수정은 별도 브랜치에서 진행 → Vercel이 자동 생성하는 Preview URL로 검증 → 문제 없으면 `main`으로 merge해 운영 배포에 반영
5. 재배포 후 동일한 시나리오로 재현 테스트

## 항목 3

### Q. HTML/CSS/JavaScript의 역할 차이를 서비스 코드 예시로 설명

- **HTML**: `index.html`의 `<input id="thought-input">`, `<button id="submit-thought-btn">`처럼 "무엇이 화면에 존재하는가"만 정의한다. 동작 로직은 전혀 없다.
- **CSS**: `style.css`의 `--accent-color`, `--bg-color` 같은 CSS 변수와 `.card`, `.nav-menu` 클래스가 "어떻게 보이는가"를 결정한다. `dark-mode.css`는 `[data-theme="dark"]` 선택자로 같은 요소의 색만 바꾼다.
- **JavaScript**: `stream.js`가 `submit-thought-btn`의 클릭 이벤트를 감지해서 입력값을 읽고(`thoughtInput.value`), Supabase에 저장하고(`TaraeStorage.saveThought`), 결과에 따라 버튼 텍스트/색을 바꾸는 "동작"을 담당한다. 즉 HTML=구조, CSS=스타일, JS=동작(상호작용)이라는 역할 분리가 코드에 그대로 드러난다.

### Q. fetch 요청이 서버리스 함수로 전달되고 응답이 돌아오는 흐름

1. 프론트에서 `fetch("/api/expand", { method: "POST", body: JSON.stringify({...}) })` 호출
2. `vercel.json`의 `rewrites` 설정(`/api/(.*)` → `/api/$1`)에 따라 Vercel이 요청을 `api/expand.py`의 `handler` 클래스로 라우팅
3. `do_POST`가 실행되며 body를 읽어 JSON 파싱 → Gemini API 호출 → 결과를 JSON으로 응답
4. 프론트의 `await response.json()`으로 결과를 받아 DOM에 반영

### Q. 환경 변수를 쓰는 이유(보안/운영)

- Gemini API 키, Supabase service role key처럼 유출되면 과금·데이터 유출로 이어질 수 있는 값을 소스코드에 직접 적으면 GitHub에 그대로 노출된다.
- 환경 변수로 분리하면 코드는 `os.environ.get("GEMINI_API_KEY")`처럼 키 이름만 참조하고, 실제 값은 Vercel 서버에만 저장되어 저장소에는 절대 남지 않는다.
- 로컬/운영 환경마다 다른 값(다른 Supabase 프로젝트 등)을 코드 수정 없이 바꿀 수 있어 운영 측면에서도 유리하다.

### Q. AI 기능을 이 서비스에 넣은 이유와 프롬프트 구성 방식

- 이유: 저장만 하는 메모 앱은 이미 많다. 타래는 "잊혀진 생각들 사이의 의외의 연결"을 사람이 직접 찾기 어렵다는 문제를 AI로 보완하려 했다 — 두 생각을 골라주면 AI가 억지로라도 연결점을 찾아 새로운 기획으로 제시한다.
- 프롬프트 구성: (1) 역할 지정("생각을 정리해주는 조력자" 등 페르소나 부여) → (2) 출력 포맷을 JSON으로 강제 지정("반드시 다음 JSON 포맷으로만 응답해") → (3) 실제 사용자 데이터(두 생각 텍스트)를 마지막에 삽입. 포맷을 강제한 이유는 Gemini의 자유 텍스트 응답을 파싱 가능한 구조로 만들기 위해서다.

## 항목 4 (심화/가정 시나리오)

### Q. 응답 지연이 잦다면(속도/비용/쿼터) 어떤 개선을 고려할까?

- 모델을 더 가벼운 버전으로 낮추거나(이미 `gemini-2.5-flash` 사용 중 — 저지연 모델), 프롬프트 길이를 줄여 토큰 수를 낮춘다.
- 프론트에서 타임아웃을 명시적으로 걸고(`AbortController`), 일정 시간 초과 시 "지금 응답이 지연되고 있어요" 안내로 전환한다.
- 자주 반복되는 요청은 캐싱(예: `daily_insights`처럼 하루 단위로 미리 배치 생성)해서 실시간 호출 빈도 자체를 줄인다.
- 비용/쿼터 초과 시를 대비해 429/쿼터 초과 응답을 별도로 감지해 사용자에게 "잠시 후 다시 시도해주세요" 같은 구분된 메시지를 준다.

### Q. AI 기능을 2개로 늘린다면 프론트/백엔드 구조를 어떻게 확장할까?

- 백엔드는 이미 `classify.py`, `connect.py`, `expand.py`처럼 기능별 파일이 분리돼 있어, 새 엔드포인트를 `api/새기능.py`로 추가하기만 하면 된다(기존 구조를 안 건드림).
- 프론트는 `storage.js`처럼 공통 API 호출 로직을 모듈로 분리해두면, 새 기능도 같은 패턴(`fetch → 로딩 표시 → 성공/실패 분기`)을 재사용할 수 있다.
- 예: 홈 화면 입력 시 `classify.py`를 호출해 실시간 태깅을 붙이는 기능을 추가하면, 기존 저장 로직 뒤에 분류 호출을 이어 붙이는 식으로 확장 가능하다(현재는 이 연결이 비어 있어 다음 개선 과제로 남겨둠).

### Q. API 키가 유출되었다면 즉시 취해야 할 조치와 재발 방지 방법은?

- **즉시 조치**: 해당 API 키를 발급처(Google AI Studio/Supabase)에서 즉시 폐기(Revoke)하고 새 키를 재발급 → Vercel 환경 변수 값을 새 키로 교체 → 재배포.
- 유출 경로가 커밋 이력이었다면, 단순 삭제로는 git history에 남으므로 `git filter-repo` 등으로 히스토리에서 완전히 제거하거나, 저장소를 새로 만드는 것도 고려한다.
- **재발 방지**: `.gitignore`에 `.env*`를 이미 등록해뒀지만, 커밋 전 `git diff --staged`로 매번 확인하는 습관, GitHub의 Secret Scanning 알림 활성화, pre-commit 훅으로 키 패턴 자동 검사 추가.

### Q. 요구사항이 "프론트 프레임워크 허용"으로 바뀐다면 장단점과 변경 범위는?

- **장점**: React/Vue 사용 시 상태 관리(현재 `TaraeStorage`를 전역 객체로 여러 파일에서 참조하는 방식)가 컴포넌트 단위 상태로 정리되고, 라우팅도 SPA 라우터로 대체해 페이지 전체 새로고침 없이 이동 가능해진다.
- **단점**: 빌드 과정(번들러)이 추가되어 배포 파이프라인이 복잡해지고, 지금처럼 파일 하나 열어서 바로 수정·배포하는 단순함이 줄어든다. 학습 목표였던 "바닐라 JS로 fetch 흐름을 직접 이해하기"와는 멀어질 수 있다.
- **변경 범위**: HTML 파일 4개를 컴포넌트로 재작성, `TaraeStorage`를 커스텀 훅/스토어로 전환, `vercel.json`의 `framework` 설정 변경, `api/` 쪽은 그대로 유지 가능(프레임워크 변경은 프론트 한정이므로 백엔드 영향은 적음).
