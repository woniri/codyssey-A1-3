## 항목 2

### Q. html/css/js/api 구조를 왜 이렇게 나눴는가?

- **HTML**은 화면의 뼈대(4개 페이지: `index.html`, `loom.html`, `vault.html`, `settings.html`)만 담당하고, 모든 페이지가 같은 `<nav>` 구조를 공유해 일관된 이동 경험을 주도록 했다.
- **CSS**는 `style.css`(공통 레이아웃/색상 변수)와 `dark-mode.css`(다크모드 전용 오버라이드)로 나눠, 라이트/다크 테마 전환이 CSS 변수만 바꿔도 전체 화면에 반영되게 했다.
- **JS**는 역할별로 분리했다: `storage.js`(Supabase CRUD를 모은 `TaraeStorage` 객체), `app.js`(다크모드 토글 등 공통 로직), `main.js`(홈 대시보드 전용 — 인사이트/태그클라우드/공명/지표), `stream.js`(홈 화면 입력 처리)로 나눠 한 파일이 여러 책임을 갖지 않게 했다.
- **api/**는 프론트와 물리적으로 분리해, 브라우저에 노출되면 안 되는 로직(Gemini API 키, Supabase service role key)을 서버에서만 실행되게 격리했다. 기능 단위(태그 분류/조합/확장/일일 인사이트/주간 리포트/테스트 발송)로 파일을 쪼개 각 서버리스 함수가 하나의 책임만 갖도록 했다.

### Q. AI 기능의 프론트 로직에서 "로딩/성공/실패" 상태를 어떻게 처리했는가?

베틀(`api/expand.py` 호출)과 타래장(`api/connect.py` 호출) 둘 다 같은 패턴을 따른다:
- **로딩**: 결과 영역을 먼저 열고 안내 문구("⚡ 4대 사유 엔진 연동 중...", "🚀 타래 내부 다중 전문가 패널을 소집하여...")를 `fetch` 호출 *이전에* 렌더링한 뒤 `await`로 응답을 기다린다. 이렇게 순서를 지켜야 브라우저가 실제로 로딩 문구를 화면에 그릴 시간을 갖는다 (처음엔 `await` 없이 동기적으로 실행돼서 로딩 문구가 아예 안 보이는 버그가 있었는데, 구조를 고치며 함께 해결했다).
- **성공**: `response.ok`이고 JSON 파싱이 되면, 정해둔 필드(`disruptor`/`architect`/`analyst`/`connector` 또는 `merged_title`/`merged_content`)를 화면에 표시한다.
- **실패**: `catch`에서 사용자에게 "🌙 지금은 ~하지 못했어요. 잠시 후 다시 시도해봐 주세요." 형태의 안내 문구를 보여준다. (초기 버전은 실패해도 하드코딩된 가짜 텍스트로 조용히 넘어가는 문제가 있어서, 실패를 있는 그대로 보여주도록 고쳤다.)

### Q. Serverless Function(Python)에서 입력 검증과 응답 포맷을 어떻게 정했는가?

- 모든 API는 `http.server.BaseHTTPRequestHandler`를 상속한 `handler` 클래스 + `do_POST`로 구현했다(Vercel Python 런타임 규격).
- 입력은 `Content-Length`만큼 body를 읽어 JSON으로 파싱하고, 필요한 키가 없으면 빈 문자열로 기본값을 준다.
- 응답은 항상 JSON + `Content-type: application/json`으로 통일했고, 에러가 나도 `try/except`로 감싸 500 상태코드와 `{"error": "..."}` 형태로 응답해 프론트가 항상 JSON을 파싱할 수 있게 보장했다.
- Gemini 응답은 자유 텍스트로 올 수 있어, 프롬프트 안에 "반드시 이 JSON 포맷으로만 응답해"라고 명시하고, 코드에서도 백틱(` ```json `) 마크다운 표시를 제거한 뒤 `json.loads`로 파싱한다.
- 실제로 겪은 사례: `api/expand.py`와 `api/connect.py`의 입출력 스키마가 프론트 호출 코드와 서로 어긋나 있었던 적이 있다(타래장이 `expand`를 부르는데 `expand`는 배열을 반환해서 프론트가 기대하던 필드가 항상 없었음). 입력/출력 스키마를 프론트-백엔드 양쪽에서 정확히 맞추는 것이 왜 중요한지 직접 겪은 사례다.

### Q. 배포 후 발생한 문제를 어떤 순서로 진단하고 수정했는가?

1. 배포(또는 로컬) 화면에서 문제 재현 → 브라우저 개발자도구(F12) → **Console** 탭에서 자바스크립트 에러 확인
2. **Network** 탭에서 관련 API 요청의 상태 코드 확인 (404 = 파일/경로 자체가 없음, 500 = 서버 코드 내부 에러, 403 = 권한/보안 정책 문제)
3. 500이면 Vercel 대시보드의 Functions 로그 또는 응답 바디의 에러 메시지로 원인 좁히기
4. 원인이 로컬에서 재현되면 로컬(`vercel dev`)에서 먼저 수정
5. 수정은 별도 브랜치에서 진행 → Vercel이 자동 생성하는 Preview URL로 검증 → 문제없으면 `main`으로 merge해 운영 배포에 반영
6. 재배포 후 동일한 시나리오로 재현 테스트

**실제로 겪었던 사례들**:
- `js/main.js` 파일이 실제로는 `api/main.js`라는 엉뚱한 위치에 있어서 대시보드 지표가 404로 로드되지 않던 문제 → `git mv`로 경로만 정정
- 타래장 AI 기능이 500 에러가 나서, Network 탭 → Response 확인 → 로컬 `.env`에 `GEMINI_API_KEY`가 비어있었던 게 원인 → 확인해보니 Vercel의 Development 환경 변수 스코프에는 값이 등록되어 있지 않았고, 심지어 "Sensitive" 변수는 Development에 아예 적용할 수 없다는 Vercel 정책 때문에 `vercel env add <KEY> development`로 별도 등록해야 했음
- 설정 페이지 저장 시 403 Forbidden → Supabase의 RLS(Row Level Security) 정책이 새 테이블에 아직 없어서 본인 데이터인데도 저장이 막혔던 것 → 정책(SELECT/INSERT/UPDATE/DELETE) 4개를 추가해서 해결
- 디스코드/슬랙 발송이 "Load failed"로 실패 → 브라우저에서 외부 웹훅으로 직접 요청하면 CORS에 막히는 경우가 있어, 서버(Python)를 한 단계 거치는 프록시(`api/test_notify.py`)를 새로 만들어 해결

## 항목 3

### Q. HTML/CSS/JavaScript의 역할 차이를 서비스 코드 예시로 설명

- **HTML**: `index.html`의 `<input id="thought-input">`, `<button id="submit-thought-btn">`처럼 "무엇이 화면에 존재하는가"만 정의한다. 동작 로직은 전혀 없다.
- **CSS**: `style.css`의 CSS 변수(`--accent-color` 등)와 `.card`, `.nav-menu` 클래스가 "어떻게 보이는가"를 결정한다. `dark-mode.css`는 같은 요소의 색만 바꾼다.
- **JavaScript**: `stream.js`가 버튼 클릭을 감지해서 입력값을 읽고 저장하며, `main.js`가 대시보드 지표·태그클라우드·인사이트를 실제 데이터로 채워 넣는 "동작"을 담당한다. 즉 HTML=구조, CSS=스타일, JS=동작(상호작용)이라는 역할 분리가 코드에 그대로 드러난다.

### Q. fetch 요청이 서버리스 함수로 전달되고 응답이 돌아오는 흐름

1. 프론트에서 `fetch("/api/expand", { method: "POST", body: JSON.stringify({...}) })` 호출
2. `vercel.json`의 `rewrites` 설정에 따라 Vercel이 요청을 `api/expand.py`의 `handler` 클래스로 라우팅
3. `do_POST`가 실행되며 body를 읽어 JSON 파싱 → Gemini API 호출 → 결과를 JSON으로 응답
4. 프론트의 `await response.json()`으로 결과를 받아 DOM에 반영

### Q. 환경 변수를 쓰는 이유(보안/운영)

- Gemini API 키, Supabase service role key처럼 유출되면 과금·데이터 유출로 이어질 수 있는 값을 소스코드에 직접 적으면 GitHub에 그대로 노출된다.
- 환경 변수로 분리하면 코드는 `os.environ.get("GEMINI_API_KEY")`처럼 키 이름만 참조하고, 실제 값은 Vercel 서버에만 저장되어 저장소에는 절대 남지 않는다.
- 실제로 Production/Preview/Development 환경마다 값을 따로 등록해야 한다는 걸 겪었다 — 운영에만 등록해두면 로컬 개발 환경에서는 값이 비어 API 호출이 실패한다. 즉 환경 변수는 "숨기는 용도"뿐 아니라 "환경별로 다른 값을 안전하게 관리하는 용도"로도 쓰인다.

### Q. AI 기능을 이 서비스에 넣은 이유와 프롬프트 구성 방식

- 이유: 저장만 하는 메모 앱은 이미 많다. 타래는 "잊혀진 생각들 사이의 의외의 연결"을 사람이 직접 찾기 어렵다는 문제(타래장의 실타래 연금술)와 "하나의 생각을 여러 관점에서 곱씹어보기 어렵다"는 문제(베틀의 4인 전문가 패널)를 AI로 보완하려 했다.
- 프롬프트 구성: (1) 역할 지정(예: "너는 4명의 서로 다른 전문가 패널이다") → (2) 출력 포맷을 JSON으로 강제 지정 → (3) 실제 사용자 데이터(선택한 생각 텍스트)를 마지막에 삽입. 포맷을 강제한 이유는 Gemini의 자유 텍스트 응답을 파싱 가능한 구조로 만들기 위해서다.

## 항목 4 (심화/가정 시나리오)

### Q. 응답 지연이 잦다면(속도/비용/쿼터) 어떤 개선을 고려할까?

- 이미 `gemini-2.5-flash`(저지연 모델)를 쓰고 있어, 프롬프트 길이를 더 줄이거나 응답 토큰 수를 제한하는 방향을 우선 고려한다.
- 프론트에서 타임아웃(`AbortController`)을 걸고, 일정 시간 초과 시 "지금 응답이 지연되고 있어요" 안내로 전환한다.
- 자주 반복되는 요청은 캐싱(예: `daily_insights`처럼 하루 단위로 미리 배치 생성)해서 실시간 호출 빈도를 줄인다.
- 429(쿼터 초과) 응답을 별도로 감지해 "잠시 후 다시 시도해주세요" 같은 구분된 메시지를 준다.

### Q. AI 기능을 2개로 늘린다면 프론트/백엔드 구조를 어떻게 확장할까?

- 실제로 이미 2개(베틀의 `expand.py`, 타래장의 `connect.py`)로 운영 중이라, 이 경험을 그대로 설명할 수 있다: 백엔드는 기능별 파일이 분리돼 있어 새 엔드포인트를 `api/새기능.py`로 추가하기만 하면 되고, 프론트는 각 페이지가 자기 영역의 fetch 로직만 갖고 있어 서로 영향을 주지 않는다.
- 다음 확장은 `classify.py`(생각 1개 → 태그/공감 문장)를 홈 화면 입력 시 실시간으로 연결하는 것이 될 예정이다.

### Q. API 키가 유출되었다면 즉시 취해야 할 조치와 재발 방지 방법은?

- **즉시 조치**: 해당 API 키를 발급처(Google AI Studio/Supabase)에서 즉시 폐기(Revoke)하고 새 키를 재발급 → Vercel 환경 변수 값을 새 키로 교체(Production/Preview/Development 전부) → 재배포.
- 유출 경로가 커밋 이력이었다면, 단순 삭제로는 git history에 남으므로 `git filter-repo` 등으로 히스토리에서 완전히 제거하거나, 저장소를 새로 만드는 것도 고려한다.
- **재발 방지**: `.gitignore`에 `.env*`를 이미 등록해뒀고, 커밋 전 `git diff --staged`로 매번 확인하는 습관, GitHub Secret Scanning 알림 활성화.

### Q. 요구사항이 "프론트 프레임워크 허용"으로 바뀐다면 장단점과 변경 범위는?

- **장점**: React/Vue 사용 시 상태 관리(현재 `TaraeStorage`를 전역 객체로 여러 파일에서 참조하는 방식)가 컴포넌트 단위 상태로 정리되고, 설정 페이지처럼 채널마다 카드가 반복되는 UI(`document.querySelectorAll` 순회)도 컴포넌트 반복 렌더링으로 훨씬 간결해진다.
- **단점**: 빌드 과정(번들러)이 추가되어 배포 파이프라인이 복잡해지고, 지금처럼 파일 하나 열어서 바로 수정·배포하는 단순함이 줄어든다.
- **변경 범위**: HTML 파일 4개를 컴포넌트로 재작성, `TaraeStorage`를 커스텀 훅/스토어로 전환, `vercel.json`의 `framework` 설정 변경. `api/` 쪽은 그대로 유지 가능(프레임워크 변경은 프론트 한정이므로 백엔드 영향은 적음).
